## 静默模式（-q）

当检测到当前处于静默模式（`-q` 参数）时：
- **不启动任何子代理**，所有审查步骤由主代理内联顺序执行
- 不输出进度提示、中间状态等非结果内容
- 仅输出最终审查报告正文

## 严重级别

| 级别 | 名称 | 说明 | 行动 |
|------|------|------|------|
| **P0** | 严重 | 安全漏洞、数据丢失风险、正确性 Bug、资源/奖励重复发放、异步操作顺序导致的数据一致性问题 | 必须阻断 |
| **P1** | 高危 | 逻辑错误、显著架构违规、性能退化 | 必须阻断 |
| **P2** | 中等 | 代码异味、可维护性问题 | 本次修复或创建跟进任务 |
| **P3** | 低风险 | 风格、命名、小建议 | 可选改进 |

## 审查流程

### 1) 预检上下文（主代理执行）

- **读取项目框架**：首先完整读取 `../skills/references/project-framework.md`，理解 Entity/Component/Service 架构、RPC 模式、异步回调、迁移机制等核心概念，作为审查的基础知识
- **优先级**：如果对话中用户提供了本地文件路径或引用，优先直接读取本地文件内容进行审查，不要走 P4 流程
- **P4 流程（无本地文件引用时）**：使用 `p4 opened -c default` 查看当前打开的文件列表，使用 `p4 diff -du` 查看具体变更内容
- 如需对比特定版本：`p4 diff2 file#have file#head` 或 `p4 diff file@=changelist`
- 识别入口点、关键路径（RPC 处理、数据库写入、Service 调用）
- 大于 500 行的 diff 按模块/功能分批审查

**输出**：将获取到的 diff 内容、相关上下文文件路径、以及 `project-framework.md` 的核心框架知识整理好，供后续子代理使用。

> **性能优化**：主代理在此步骤中**一并读取所有检查清单文件**（`solid-checklist.md`、`lua-code-style.md`、`system-dev-checklist.md`、`code-quality-checklist.md`），将内容直接内联到子代理 prompt 中，避免子代理再执行 `read_file` 调用。

### 2) 审查扫描（2 个子代理并行执行）

将审查拆为 **2 个子代理**执行，每个子代理接收：
- **项目框架上下文**：`project-framework.md` 的完整内容（Entity/Component/Service 架构、RPC 模式、异步回调约定等），确保子代理理解项目结构
- 完整的 diff 内容
- 需要审查的文件路径列表
- **内联的检查清单内容**（主代理在步骤 1 已读取，直接嵌入 prompt）

> **并行执行（强制）**：必须在**同一个 tool call 块**中同时发起 2 个子代理调用，使其被框架并行调度。严禁串行调用（先等子代理 1 返回再调子代理 2）。

每个子代理独立输出该维度的发现项列表，格式为：`[P级别] [file:line] 标题 — 描述 + 建议修复`。

#### 子代理 1：架构设计 & 代码风格

内联检查清单：`solid-checklist.md` + `lua-code-style.md` 的内容

扫描要点（架构与设计原则）：
- **SRP**：单个组件/Service 是否承担了不相关的职责
- **OCP**：新增行为是否需要频繁修改已有 if/switch 逻辑
- **DIP**：业务逻辑是否直接耦合底层实现
- 代码异味：过长函数、特征嫉妒、数据泥团、霰弹式修改等

扫描要点（代码风格）：
- **命名规范**：变量/函数/类是否符合驼峰命名约定，kg_require 模块名是否使用小写下划线
- **格式规范**：字符串是否使用双引号、table 初始化是否规范、if 语句是否有多余括号
- **Hotfix 兼容**：是否有不利于热更的写法（模块级 local 函数、table 持有 function、模块级执行逻辑）
- **日志规范**：是否使用 `logInfoFmt` 等 format API 而非字符串拼接，格式符是否统一使用 `%s`
- **性能习惯**：循环内是否缓存了频繁访问的表字段，是否有不必要的临时 table/闭包创建
- **常见陷阱**：table 判空是否使用 `next`、条件判断是否显式比较、遍历中删除元素是否安全
- **Entity 引用**：业务模块是否持有了 Entity 引用而非 ID
- **pcall 使用**：是否使用了 `pcall`（应改为 `custom_xpcall`）

#### 子代理 2：C7 项目专项安全 & 代码质量

内联检查清单：`system-dev-checklist.md` + `code-quality-checklist.md` 的内容

扫描要点（C7 项目专项 & 安全可靠性）：
- **空值安全**：是否有未检查 nil 的地方（Lua 中 nil 访问易导致静默失败）
- **参数检查**：Exposed RPC 入口是否缺少参数校验；客户端上行参数禁止 None 类型
- **MongoDB 安全**：是否有空字符串 key 写入风险；字段名禁止包含 `.`
- **网络协议一致性**：RPC 参数是否与 `NetDefs/*.xml` 定义一致
- **日志规范**：是否使用了 `LOG_INFO`/`LOG_ERROR`/`LOG_WARNING` 而非 `print`
- **数据表引用**：是否通过 `TableData` 接口访问数据，而非直接 `require`；禁止代码中缓存 data 数据；禁止遍历整张表
- **内存泄漏**：是否有未清理的定时器或回调注册；定时器不超过 1 天，在日刷新处检查设置
- **并发安全**：异步操作（Service 调用回调）后是否检查了对象有效性
- **异步操作顺序**：连续调用的多个方法是否都保证同步执行，回调链是否被意外简化
- **属性安全**：上线后禁止修改属性数据类型或删除存盘属性
- **经济安全**：购买必须先扣币再发道具；发放奖励必须先标记再发放；所有投放必须限次
- **RPC CD 限制**：大压力 RPC 必须加协议级 CD
- **幂等性**：重要操作 RPC 须执行前检查保证幂等
- **系统开关**：每个业务系统必须有开关
- **时间兼容**：涉及时间节点的系统须兼容时间错过的情况
- **持久化**：重要操作后手动触发写库；禁止通过 MongoDB API 直接修改 entity 数据
- **资源耗尽**：无界循环、无超时的外部调用、无限增长的缓存
- **敏感数据**：日志中是否泄露玩家敏感信息
- **外部服务**：调用须容错 + 报警 + 本地兜底方案

扫描要点（代码质量）：
- **错误处理**：是否吞没异常、是否缺少错误处理、异步错误是否传播
- **性能**：N+1 查询、热路径高开销、缺失缓存；高频操作谨慎创建 table；批量删除用 `batchremove`
- **边界条件**：nil 处理、空表处理、数值边界、off-by-one 错误
- **死代码**：未使用的函数、废弃的功能开关
- **数据库操作**：查询须有索引；禁止 `skip`；排序须有索引

### 3) 结果汇总与去重（主代理执行）

收集 2 个子代理的输出后：

1. **去重**：两个子代理可能发现同一问题（如 nil 安全同时被子代理 1 和 2 发现），合并为一条，保留最高严重级别
2. **统一编号**：按 P0 → P1 → P2 → P3 顺序重新编号
3. **交叉检查**：基于两个子代理的发现，识别跨维度的系统性问题

### 4) 输出格式

```
## 代码审查摘要

**审查文件数**：X 个文件，Y 行变更
**总体评估**：[通过 / 需要修改 / 仅建议]


## 发现项

### P0 - 严重
（无 或 列出）

### P1 - 高危
1. **[file:line]** 简要标题
   - 问题描述
   - 建议修复

### P2 - 中等
2. （继续编号）

### P3 - 低风险

## 清理/迭代计划
（如适用）

## 其他建议
（可选改进，不阻断）
```

### 5) 导出审查记录

将审查结果导出为 markdown 文件，保存到 `Server/docs/reviews/` 目录下作为历史记录（基于 workspace 中 `Server/` 的实际路径定位）。

**文件命名**：`CR-<日期>-<来源标识>.md`
- 日期格式：`YYYYMMDD`
- 来源标识：
  - P4 changelist：CL 编号，如 `CR-20260318-CL123456.md`
  - 多个 CL：用下划线连接，如 `CR-20260318-CL123456_123457.md`
  - 本地文件：主文件名（不含路径和扩展名），如 `CR-20260318-BagComponent.md`
  - default changelist：`default`，如 `CR-20260318-default.md`

**文件内容**：

```markdown
# Code Review — <来源描述>

**日期**：YYYY-MM-DD
**审查来源**：<CL 编号 / 文件路径列表>
**审查文件数**：X 个文件，Y 行变更
**总体评估**：[通过 / 需要修改 / 仅建议]

---

（完整的步骤 4 输出格式内容）
```

**注意**：
- 目录不存在时自动创建
- 同名文件存在时追加序号后缀（如 `CR-zhangsan-20260318-CL123456-2.md`）
- 导出文件**不纳入版本控制**（不执行 `p4 add`），仅作为本地历史记录

### 6) 后续确认

展示发现后，询问用户如何处理：
1. **全部修复** - 实施所有建议
2. **仅修复 P0/P1** - 处理严重和高危问题
3. **修复指定项** - 用户指定修复哪些
4. **不做修改** - 审查完成

**重要**：未经用户确认，不要主动实施修改。
