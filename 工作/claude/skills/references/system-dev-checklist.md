# 系统开发 Checklist

## 属性 (Entity Def)

- 上线后严禁修改原有属性的数据类型，merge 生产分支时需检查 XML Diff
- 重构时须考虑历史数据兼容（旧 DB 记录可能缺少新增字段）
- 上线后存盘属性禁止删除（即使系统废弃），避免重名属性类型冲突导致旧角色初始化失败
- 属性数据必须测试量级；大部分时间不活跃的系统数据不宜直接放到玩家身上，应单独写库
- 系统内部 bool 属性使用 `BoolPropDefs` 定义，不直接加到 xml
- 系统内部数据类型用 `DataTypes` 定义在自己的 xml 中，减少 `alias.xml` 体积和冲突

## 组件 (Component)

- 禁止在组件初始化时执行过多逻辑（含 RPC 同步）；非核心系统使用异步初始化，大部分数据在客户端打开 UI 时才请求
- 每个业务系统必须有开关，支持设置后同步客户端，用于风险控制和 AB 测试
- 涉及随机产出的逻辑（抽奖、掉落等）须独立并提供 QA 测试接口

## 持久化

- 重要操作后（升级、充值、高价值交易）手动触发玩家数据写库，降低回档概率；但不可滥用（低等级升级、开服高峰等不触发）
- 业务代码严禁通过 MongoDB API 直接修改 entity 数据，统一使用引擎 saveTo 接口
- 禁止将配置表数据作为属性持久化（数据膨胀 + 单 doc 溢出风险 + 策划改表难处理）

## RPC

- 暴露给客户端的RPC，每个参数务必需要做检查
- 暴露给客户端的大压力 RPC 必须在协议上增加 CD 限制：异步操作 0.5-1 秒，数据库/广播类 3 秒以上
- 客户端上行 RPC 参数禁止使用 None 类型
- 单个 RPC 参数数据量不能过大，大量数据必须分包/分页拉取
- 数据同步尽量用增量 RPC（首次全量推送后，后续只推变更）

## 数据库

- DB 查询必须有对应索引，建索引时须考虑未来分片
- 查询禁止使用 `skip`
- 有排序需求时检查是否有可用索引
- 非高时效查询设置合理的 `read_preference`
- 每一处独立写库逻辑必须记录并组内 Review（特别是 update/delete 操作）
- 频繁读操作使用缓存（DataCenterService、Redis 等）
- 数据库字段名禁止包含 `.`（MongoDB 限制）
- 能 batch 的操作使用 bulk 接口，禁止 for 循环逐条处理
- 大数据量初始化分批分帧读取，按索引和 shardkey 设计分批方式

## 数值 / 经济安全

- 不可挽回的数据错误时直接 `error` 中断执行，禁止自动修正数据（如扣币大于持有量时禁止取 min）
- 涉及玩家重要数值修改必须有完整 SA Log（修改前、修改后、修改来源）
- 一笔完整交易的 SA Log 必须记录唯一 uuid，用于串联多条日志和对账
- 新系统涉及投放必须有对应新渠道，不得借用其他渠道
- **购买操作必须先扣币再发放道具**
- **所有投放必须使用限次奖励**，防止 bug 被刷时损失可控
- **发放奖励必须先设置标记再发放**，避免异常导致刷奖励

## 异步

- 重要数据修改需确保异步执行成功，且必须有容错
- 注意异步 RPC 死循环（引擎层已支持检测）
- 重要操作的 RPC 需保证幂等（执行前检查，如摆摊、活动开启等）

## 外部服务

- 调用外部服务必须容错，中断操作进行报警，给玩家合适提示，部分情况需本地方案兜底
- 调用地址必须配置在 config 中，不可写死
- 需考虑外部服务回档时的处理预案（外部服务通常不可回档）
- 外部服务字段必须检查大小写和命名差异（不同 API 版本可能字段命名不一致）
- 外部回调函数尽量做到幂等

## 时间 / 定时器

- 不要设置超过 1 天的定时器，在日刷新处做检查和设置
- 涉及多时间节点的系统须经 QA 改时间自然跑完的测试流程
- 涉及时间节点的系统必须兼容时间错过的情况（维护、异常重启等），需设置执行标记 + 起服时间检查
- 服务器时间须同步到客户端（倒计时、拍卖、摆摊等依赖双端时间一致性和时区）

## 策划配置表

- 配置中的时间字符串必须在导表后处理中导出为时间戳（runtime 实时转化性能差）
- 禁止在代码中遍历整张表，使用后处理导出索引表
- 禁止代码中缓存 data 数据（热更容易遗漏）
- data 数据只读化限制（避免意外修改 table）
- 不允许使用 `loadstring` 处理数据表中的公式/函数，导表时直接导出为函数

## 性能

- 高频操作谨慎创建新 table
- 批量删除用 `table.batchremove`/`table.multiremove`，禁止循环 `table.remove`
- 避免批量往 list 中间增加元素
- 元素增删查优先 map 操作（`t[k]=v`/`t[k]=nil`），需要顺序/数量才用 list
- 不影响热更时，高频访问处加 local 缓存（每一次 `.` 操作都有开销）
- Service 中批量发奖/耗时 RPC 必须分帧，避免人数过多导致卡顿
- 集中进入的多人活动场景需做前后端扰动（随机延迟入口显示/请求处理），500 人以上必须有进入队列
- 玩法中的匹配需求必须有匹配队列及上限控制，每个队列有开关可调速
