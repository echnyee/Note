# SOLID 原则检查清单（C7 服务端适配版）

## SRP（单一职责）

- 组件是否承担了不相关的职责（如 HTTP + DB + 业务规则混在一个组件中）
- 大型组件/Service 内聚性低，有多个变更原因
- 函数编排了多个不相关的步骤
- 全知对象：对系统了解太多
- **自问**："这个模块因什么单一原因而变更？"

### C7 特定场景
- 单个 Component 文件是否同时处理了多种不相关的 RPC 请求
- Service 是否混入了应该属于 Component 的 Actor 级逻辑
- 子模块拆分（如 `_Attr`、`_Apply`）的职责是否清晰

## OCP（开放封闭）

- 添加新行为需要修改多处 switch/if 分支
- 功能增长需修改核心逻辑而非扩展
- 缺少插件/策略/Hook 点来处理变化
- **自问**："能否不修改已有代码就添加新变体？"

## LSP（里氏替换）

- 子类检查具体类型或对基类方法抛出异常
- 重写方法弱化了前置条件或加强了后置条件
- 子类忽略或空实现父类行为
- **自问**："任何子类替换后调用方是否无感知？"

## ISP（接口隔离）

- 接口方法众多，大多数实现者未使用
- 调用方为狭窄需求依赖宽接口
- 接口方法有空/桩实现
- **自问**："所有实现者是否使用了所有方法？"

## DIP（依赖反转）

- 高层逻辑依赖具体的 IO、存储或网络类型
- 硬编码实现而非抽象或注入
- 导入链将业务逻辑耦合到基础设施
- **自问**："能否不修改业务逻辑就替换实现？"

## 常见代码异味

| 异味 | 表征 |
|------|------|
| **过长函数** | 函数 > 30 行，多层嵌套 |
| **特征嫉妒** | 方法使用其他类的数据多于自身 |
| **数据泥团** | 相同参数组反复传递 |
| **基本类型偏执** | 用字符串/数字代替领域类型 |
| **霰弹式修改** | 一个变更需跨多文件修改 |
| **发散式变更** | 一个文件因多个不相关原因变更 |
| **死代码** | 不可达或从未调用的代码 |
| **投机泛化** | 为假设的未来需求创建抽象 |
| **魔法数字/字符串** | 硬编码值未使用命名常量 |

## 重构启发

1. **按职责拆分，而非按大小** — 小文件也可能违反 SRP
2. **按需引入抽象** — 等待第二个使用场景
3. **保持增量重构** — 先隔离行为再移动
4. **先保持行为** — 重构前先加测试
5. **按意图命名** — 命名困难说明抽象可能有误
6. **组合优于继承** — 继承创建紧耦合
7. **让非法状态不可表示** — 用类型来强制约束
