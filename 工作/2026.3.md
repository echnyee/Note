## 10
#mongo 
### skip
分页查找，假设我使用Mongo的skip来分批加载数据，例如第一次加载1-100，第二次加载101-200，第三次加载201-300，那第二次的时候要读200条数据，第三次要读300条，此时Mongo才知道应该从哪里开始skip，所以当次数变多时，skip的效率会大幅度下降。
原因是：
- B-Tree 索引只知道 **顺序**
- 不知道“第 N 条记录的位置”

所以数据库只能：
从起点顺序遍历
丢弃前面的数据。

db.col.find({status:"ok"}).skip(200).limit(100)
加过滤条件的话也不一定解决问题，如果status没有索引，全表扫描，性能更差，如果有索引，会好一点，但也是在小规模下不同的重复，依然会有性能问题

### cursor
**一次 `find()` 查询通常会创建一个 cursor**，它本质上是一个**服务器端的查询结果迭代器**。
**Cursor = 查询结果的“游标 / 迭代器”**
当你执行：
```
db.col.find({score: {$gt:100}})
```
MongoDB 不会一次把所有数据返回给客户端，而是：
- **创建一个 cursor**
- 先返回 **第一批数据（first batch）**
- 客户端继续请求 → 返回下一批

```
db.col.find().batchSize(100)
```

意思是：
每次返回 100 条
但：
⚠️ **cursor 仍然是一个查询，只是分批返回**
这里cursor是一次完整的返回，不可终端，和我之前的分页本质区别就是是否可打断。

#mongo #数据库 
一些关系型和非关系型的思考
MySQL，表结构比较固定，新增结构会更复杂，一些复杂数据，会使用外键去进行与外部的链接
MongoDB，key-value或者说json格式，更灵活，改动结构时不需要去改表scheme，因为也没有这个东西，一般不会有join操作，每次查询都是单表查询，这样对分布式非常友好。对比MySQL，有join的情况如果2个表在分布式集群的不同进程下，可能还会有跨进程的通信。会将这一些操作放在业务层进行，例如玩家正常进来加载自己的数据，然后在后续某一步懒加载邮件的数据
