## 10
#mongo 
### skip
分页查找，假设我使用Mongo的skip来分批加载数据，例如第一次加载1-100，第二次加载101-200，第三次加载201-300，那第二次的时候要读200条数据，第三次要读300条，此时Mongo才知道应该从哪里开始skip，所以当次数变多时，skip的效率会大幅度下降。
原因是：
- B-Tree 索引只知道 **顺序**
- 不知道“第 N 条记录的位置”

所以数据库只能：
从起点顺序遍历
丢弃前面的数据。

db.col.find({status:"ok"}).skip(200).limit(100)
加过滤条件的话也不一定解决问题，如果status没有索引，全表扫描，性能更差，如果有索引，会好一点，但也是在小规模下不同的重复，依然会有性能问题

### cursor
**一次 `find()` 查询通常会创建一个 cursor**，它本质上是一个**服务器端的查询结果迭代器**。
**Cursor = 查询结果的“游标 / 迭代器”**
当你执行：
```
db.col.find({score: {$gt:100}})
```
MongoDB 不会一次把所有数据返回给客户端，而是：
- **创建一个 cursor**
- 先返回 **第一批数据（first batch）**
- 客户端继续请求 → 返回下一批

```
db.col.find().batchSize(100)
```

意思是：
每次返回 100 条
但：
⚠️ **cursor 仍然是一个查询，只是分批返回**
这里cursor是一次完整的返回，不可终端，和我之前的分页本质区别就是是否可打断。

## MySQL
表结构比较固定，新增结构会更复杂，一些复杂数据，会使用外键去进行与外部的关联
这里需要注意，外键和join有关联，但不是一回事。
## 外键
以玩家与邮件为例（MySQL 表达）：
- `players(id, ...)` 主表（被引用）
- `mails(id, player_id, ...)` 子表（引用别的表）
如果你在 `mails.player_id` 上建外键：
- **含义**：`mails.player_id` 的值必须出现在 `players.id` 中
- **效果**：
    - 插入一封邮件，`player_id=123`，但 `players` 里没有 `id=123` → 插入会失败
    - 删除一个玩家，如果还有邮件引用它：
        - 默认会阻止删除（RESTRICT / NO ACTION）
        - 或者你设置了级联删除（ON DELETE CASCADE）就会自动把邮件删掉
        - 或设置 ON DELETE SET NULL 则把 `player_id` 置空（游戏里一般不这么做）

**外键的核心是：防止“孤儿记录”（orphan rows）**，比如存在 `mail.player_id=123` 但玩家 123 根本不存在。
> 注意：很多游戏项目不建外键，是工程取舍（性能、分库分表、写入路径可控），但“不建”不代表它不是外键关系——只是你没有让数据库强制它。‘

这里外键有一个数据库的概念，主要就是为了防止孤儿记录，另外还有一个逻辑上的外键概念，就是本质上他们是有关联的，例如邮件表里的playerId和真正的playerId，但是数据库上是否建了这个外键就看项目需要。可以明显的看到有外键的情况下数据一致性会更高。但是对于分布式来说很不友好，例如玩家分表了，那么在其他地方有建外键，如果和这张玩家表不在一个shard，那就需要去其他shard节点来做外键的一致性（可能跨进程）
## join
另外关系型数据库支持join，join对于分布式也一样不友好，因为涉及多表的数据协同，当然业务层可以不用join，但是这就丢失了关系型数据的一个特性。
另外join在业务层也能做，但是和在db层会有差别
在数据库做join有很多优化的手段，数据在 DB 进程内（甚至在 buffer pool / page cache 内）就能完成关联与过滤，**不需要把大中间集传出 DB**；通常只把最终结果集返回。

在应用层也可以做类似join的事情。
但结合前面的运营筛选玩家例子：为什么 DB JOIN 往往更合适？
需求：找出“最近 7 天登录过、等级 ≥ 30、且购买过某礼包、且当前还在某公会”的玩家列表，用于发奖/投放。
运营筛选常见特点：
- 条件多且组合变化快
- 可能需要输出的是“玩家列表 + 分页 + 排序”
- 候选集可能很大（全服级别）

这时如果用应用层 join，很容易出现：
- 先从 orders 找出几十万 player_id
- 再去 players 过滤等级
- 再去 guild_members 过滤公会
- 中间集合巨大、网络和应用内存爆

而 DB 侧可以：
- 用索引把每个表的候选集压到最小
- 用 EXISTS/半连接减少结果爆炸
- 直接在 DB 里排序/分页，应用拿一页结果就行

**区别在于：**
- DB 中间结果更多是在 DB 机器内部流动/落盘（可被 DB 优化/限制）
- 应用层中间结果往往会跨网络、占应用内存，并把压力扩散到更多层
## MongoDB
key-value或者说json格式，更灵活，改动结构时不需要去改表scheme，因为也没有这个东西，没有外键，也不会有join操作，每次查询都是单表查询，这样对分布式非常友好。一般会将这一些操作放在业务层进行，例如玩家正常进来加载自己的数据，然后在后续某一步懒加载邮件的数据

#分布式 
一般一致性hash其实很少见，更常见的是预分配，先分配好1024个逻辑bucket，有一个管理者保存每个节点负责的bucket，进行扩展时管理者从现有的每个节点负责的bucket里面抽几个放到新节点里面。
