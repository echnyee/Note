## branch
```
ycy19@DESKTOP-Q0MP5EC MINGW64 /d/工作/git/snake (master)
$ git branch -a
* master
  remotes/origin/HEAD -> origin/master
  remotes/origin/master
```
这里每一行解释
\* master本地branch，\* 表示当前在这个branch
remotes/origin/HEAD -> origin/master 表示远程默认的branch是master
remotes/origin/master  是本地的remote-tracking branch，是一个本地对于remote branch的快照
可以把它理解成：
> “我本地上次已知的远程 `origin` 仓库里的 `master` 分支位置”
### 例子
一开始git clone之后，本地会创建本地的master branch，还有本地的remote-tracking branch，也就是origin/master，此时master默认是tracking origin/master的
```
A---B---C   master, origin/master
```
之后远程多了一个提交D，本地没有fetch之前，是不会有变化的
```
git fetch
```
之后会变成这样
```
A---B---C   master
         \
          D   origin/master
```
```
git merge origin/master
```
之后会变成
```
A---B---C---D   master, origin/master
```
如果使用git pull的话其实就是git fetch + git merge origin/master，这里的origin/master是通过tracking的信息自动补上的
一般来说会使用git pull --rebase（等于git fetch + git rebase origin/master）
### rebase
如果本地有提交，不使用rebase会导致提交历史变成非线性的，提高阅读成本
`rebase` 可以理解成：**变基**，也就是“换个基底重新接提交”。
比如现在情况是：
远程：
bash
```
A---B---C  origin/main
```
你的本地：
bash
```
A---B---D---E  main
```
你本地有两个提交 `D` 和 `E`，但远程已经多了一个 `C`。

如果你执行：
bash
```
git pull --rebase
```
Git 会做这件事：
1. 先把你的本地提交 `D`、`E` 临时拿下来
2. 把远程的 `C` 更新到本地
3. 再把 `D`、`E` 重新应用到 `C` 后面
结果变成：
bash
```
A---B---C---D'---E'
```
注意：
- `D'` 和 `E'` 是重新生成的提交
- 提交内容可能一样
- 但 commit hash 会变

## 常见的通过etcd实现的服务注册/发现
大致步骤
### etcd 中的数据模型（建议）注册方或者说service提供者

- 前缀：`/grpc/<service-name>/`
- 实例 key：`/grpc/<service-name>/<instance-id>`
- value（JSON/Proto 都行）包含：
    - `addr`: `"10.0.1.23:50051"`
    - 可选 `metadata`：`weight/tags/zone`

注册时：`Put(key,value, WithLease(leaseID))`，并 `KeepAlive`。
### gRPC 客户端侧：Resolver 做 “Get + Watch”

#### 核心流程

1. **初始拉取**：`Get(prefix, WithPrefix())` → 得到全量实例 + revision
2. **建立 watch**：`Watch(prefix, WithPrefix(), WithRev(rev+1))`
3. 维护本地 `map[instanceID]addr`
4. 每次变更后调用 `cc.UpdateState(resolver.State{Addresses: ...})`
5. gRPC LB 策略根据 addresses 做连接与选择

这样能做到：

- 新实例上线：watch 收到 PUT → addresses 增加 → 客户端自动连上新后端
- 实例下线（主动删除或 Lease 过期）：watch 收到 DELETE → addresses 删除 → gRPC 会关掉/迁移连接

> 关键点：**先 Get 再从对应 revision 开始 Watch**，避免漏事件。
### 负载均衡与连接行为要注意的点

- **round_robin**：会在多个 SubConn 间轮询（适合多实例）
- **pick_first**：只选第一个地址（更像主备，不适合水平扩展）
- 地址列表变化时，gRPC 会增删 SubConn；但连接迁移不是“瞬时无损”，要配合：
    - 重试/超时
    - 服务端优雅停机（先停止接新请求，再等 in-flight 完成，再撤销注册）
