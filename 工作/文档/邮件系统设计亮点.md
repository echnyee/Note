# 邮件系统设计亮点分析

## 一、数据一致性：先 DB 后内存 + CAS 防并发

附件提取是邮件系统最容易出问题的环节，这里的设计相当严谨。

### 1.1 先写 DB，回调确认后才改内存

`extractMail` 不直接修改内存中的 `mail.isExtracted`，而是先通过 `extractMailUpdate` 写 DB，在 `extractMailCallback` 回调确认成功后才更新内存状态。这样即使 DB 写入失败，内存不会出现脏数据。

### 1.2 数据库级 CAS 防重复领取

DB 层用条件更新做天然的并发控制：

```lua
-- MailComponent.lua extractMailUpdate
["$or"] = {
    { isExtracted = false },
    { isExtracted = { ["$exists"] = false } },
}
```

两个请求同时提取同一封邮件，只有一个能 match 成功，另一个 `matchedCount=0`，回调中据此判断并发冲突。本质上是数据库级别的 CAS（Compare-And-Swap）操作。

### 1.3 全服邮件的乐观标记 + 失败回滚

`checkServerMail` 先乐观标记 `receivedServerMailIds` 防重复收取，通过 `DoPersistent` 确保存盘。存盘失败时精确回滚内存标记，保证下次能重新收取这批邮件。

### 1.4 全服邮件 serverMailId 的原子自增

`serverMailId` 通过 `SHARED_COUNTER` 集合中的 `server_mail_id` 计数器生成，而非使用 `SERVER_MAIL` 集合自身的 `_id`：

- MongoDB 默认 `_id` 是 ObjectId（字符串），不是递增数字，无法直接做大小比较
- `IncAndFetchSharedCounter` 使用 `MongoFindAndUpdate` + `$inc` 原子操作，多进程并发发送全服邮件时保证唯一且递增
- 先拿号再写入的流程比写入后取号更可控，`serverMailId` 在写 DB 前就已确定，后续广播通知也直接携带这个值

### 1.5 receivedServerMailIds 逐条记录而非水位线

玩家使用 `receivedServerMailIds`（map<number, boolean>）逐条记录已接收的全服邮件，而非单一水位线 `lastServerMailId`。

全服邮件存在条件过滤（等级/建号时间/生效时间/黑名单），玩家可能暂时不满足某封邮件的条件。假设邮件序列为 1,2,3,4,5，玩家当前只满足 1,3,5 的条件，若用水位线 `lastServerMailId=5` 一刀切，后续等级提升后 2 和 4 就永远收不到了。逐条记录方式下，跳过的邮件不会被标记，下次 `checkServerMail` 时重新检查条件，满足了就能收到。

这个设计用少量的存储开销（每个玩家多存几个 number key）换来了"条件延迟满足"的能力。

### 1.6 发送邮件的去重处理

在 processInsertMailList 中的效果就是幂等写入：

```
{
selector = { receiver = mail.receiver, id = mail.id },
update = { ["$setOnInsert"] = mail, ["$currentDate"] = {insertTS=true} },
upsert = true,
}
```

用 receiver + id 做唯一匹配：

第一次写入 → 文档不存在，插入完整邮件数据
重复写入（比如网络抖动导致重试）→ 文档已存在，直接跳过，不会覆盖任何字段

这种"乐观标记 + 持久化确认 + 失败回滚"的模式贯穿整个系统。

## 二、容错设计：三路分支 + 退回机制

### 2.1 单封提取的三路分支

附件提取的结果处理覆盖了所有边界情况：

- 全部进包 → 正常完成，通知客户端
- 部分进包 → 放不下的物品通过 `_sendRemainderMail` 以新邮件退回，玩家不会丢东西
- 全部放不下 → 回滚 DB 状态（`isExtracted` 恢复为 false），邮件恢复为未领取

### 2.2 退回邮件使用专用模板

```lua
-- _sendRemainderMail 使用专用模板 EXTRACT_FAIL_MAIL_ATTACHMENT_TEMPLATE_ID
Game.Process:sendAvatarActorSystemMail(
    opNUID, self.LogicServerID, self.id,
    mail_const.EXTRACT_FAIL_MAIL_ATTACHMENT_TEMPLATE_ID,
    item_const_source.ITEM_SOURCE_MAIL,
    nil, remainderItems
)
```

走标准的 `sendAvatarActorSystemMail` 流程，退回邮件也享受队列合并、批量写入等优化。

### 2.3 批量提取的贪心策略

`extractAllAttachments` 不是简单的"全能放就全放，否则全不放"，而是贪心策略：

- 从旧到新逐封累积检查背包容量
- 能放下的确认提取，放不下的加入回滚列表
- `tempRewards` 做累积检查，失败时回退到上一封成功时的深拷贝快照
- 最终对回滚列表逐封恢复 DB 状态

这对玩家体验很友好——尽可能多地领取能放下的邮件。

## 三、性能优化：微批处理 + 懒加载 + 内存缓存 + 分帧广播

### 3.1 发送队列的微批处理

```lua
-- addSendAvatarActorMailTask: 加入队列，0.1s 后统一处理
table.insert(self.sendAvatarActorMailTaskQueue, {logicServerID, avatarId, mail, logId})
if not self.mailSendTimer then
    self.mailSendTimer = self:addTimerEx(0.1, false, "processAvatarActorMailTask")
end
```

- 0.1 秒内的所有邮件发送请求合并为一次 DB 操作
- 消费时先 swap 队列引用，新任务进新队列不干扰当前批次
- 按 `logicServerID` 分组后每 100 封一批 `MongoBulk` 写入
- `$setOnInsert` + `upsert=true` 保证幂等，重复写入不会覆盖
- `table_new(BATCH_SIZE, 0)` 预分配数组空间，减少 Lua table rehash

### 3.2 邮件列表懒加载

玩家登录时不加载邮件列表，首次打开邮件界面才从 DB 拉取：拉取时会检查bNeedQueryMail 标记，如果有缓存且无变化的情况下不会拉取，
为 false 则触发一次 DB 查询。

```lua
function MailComponent:getMailList()
    if self.mailList == nil then
        self.needQueryMail = true
    end
    if self.needQueryMail then
        self:loadMailsAndCall("sendMailBriefList", Game.EmptyTable)
    else
        self:sendMailBriefList()
    end
end
```

`loadMailsAndCall` 封装了"加载后回调"模式，��调用方完全透明。收藏夹 `mailFavs` 也采用同样的独立懒加载。

### 3.3 清理防抖

```lua
function ProcessComponent_Mail:ScheduleCleanOverLimitMails(logicServerID, userID)
    if self.cleanTimer[userID] then return end
    self.cleanTimer[userID] = true
    self:addTimerEx(1, false, "cleanOverLimitMails", logicServerID, userID)
end
```

per-user 防抖，1 秒内多次触发只执行一次清理。清理查询用 `projection = {id: 1}` 只取 id 字段，减少网络传输。

### 3.4 全服邮件内存缓存（Game.Servermails）

全服邮件是典型的"读多写少"场景——写入频率低（运营操作），但读取频率高（每个玩家登录/上线都要检查）。

`BatchLoadServerMail` 把全服邮件从 DB 加载到进程内存 `Game.Servermails` 中：

```lua
-- 数据结构：按逻辑服分组，serverMailId 做 key
Game.Servermails = {
    [logicServerID] = {
        [serverMailId] = mailInfo,
        ...
    }
}
```

- 玩家的 `checkServerMail` 直接读内存，零 DB 开销，把 N 个玩家的 N 次 DB 查询降为 0
- 按 `logicServerID` 分组 + `serverMailId` 做 key，查询都是 O(1)
- 只加载未过期邮件（`destroyTime > now`），内存占用可控
- 用游标分页递归加载（每次 limit 条，取最后一条 serverMailId 作为游标），避免单次查询数据量过大

### 3.5 分帧广播通知在线玩家（BroadcastToLocalAllAvatars）

全服邮件到达后需要通知所有在线玩家检查，`BroadcastToLocalAllAvatars` 采用分帧处理避免 CPU 尖峰：

```lua
local batchSize = 100  -- 每批 100 个玩家
for batchIndex = 1, batches do
    Game.Process:addTimer((batchIndex - 1) * 0.1, false, function()
        for i = startIdx, endIdx do
            local avatar = _script.entities[avatarIDs[i]]
            if avatar and not avatar.isDestroyed then
                safeCall(avatar[methodName], avatar, unpack(args, 1, argsCount))
            end
        end
    end)
end
```

三层保护：
- 每批 100 人，批次间隔 100ms，分散 CPU 负载
- `safeCall` 包裹每个玩家的调用，单个玩家异常不影响��他玩家
- `not avatar.isDestroyed` 检查防止处理已销毁实体

## 四、并发控制：提取锁

```lua
function MailComponent:setExtractMailLock()
    if self.extractMailLock == true then return false end
    self.extractMailLock = true
    self.extractTimerID = self:addTimerEx(10, false, "unsetExtractMailLock")
    return true
end
```

- 布尔标记做轻量互斥，防止玩家快速点击导致 `extractMail` 和 `extractAllAttachments` 并发执行
- 10 秒超时定时器作为兜底，即使 DB 回调异常也不会永久死锁
- 解锁时同步清理定时器，避免泄漏

简单、有效、安全。

## 五、清理策略：精细化保护

`cleanOverLimitMails` 的查询条件设计细致：

```lua
{
    ["receiver"] = userID,
    ["addToFav"] = { ["$ne"] = true },        -- 待收藏的不参与清理
    ["$or"] = {
        { ["type"] = MAIL_TYPE_TEXT },          -- 纯文本邮件
        { ["type"] = MAIL_TYPE_ATTACHMENT, ["isExtracted"] = true }  -- 已提取的附件邮件
    }
}
```

- 未提取附件的邮件不会被清理（保护玩家利益）
- `addToFav=true` 的邮件排除在外
- 按 `createTime` 降序，超出上限的从最旧的开始删除

## 六、触发邮件：条件解耦 + 引擎约束感知

### 6.1 静态条件与动态条件解耦

`_canTriggerMailActivate_EXCLUDE_TRIGGER_CONDITION` 检查静态条件（时间/等级/建号时间），与动态触发条件（TriggerNumber）分开处理，职责清晰。

登录时做一次全量扫描，将触发邮件分为三类：
- 已满足 → 直接发送
- 未满足但未过期 → 注册触发器等待
- 无条件 → 直接发送

### 6.2 定时器的引擎约束感知

```lua
if earliestSendTime - Game.TimeInSecCache > 3600*24*30 then
    goto continue  -- 超过30天（引擎timer上限），跳过
end
```

- 考虑了引擎 timer 的 30 天上限约束，超出范围的不注册
- 已过期的直接跳过，不浪费 timer 资源
- 重入安全：先删旧 timer 再注册新的，支持热更新配置后重新 setup
- `onTriggerMailTimer` 触发时用 `safeCall` 保护，单个玩家异常不影响其他玩家

## 七、邮件黑名单：双维度 + 跨进程同步

### 7.1 双维度封禁

支持按 `templateId` 和 `opNUID` 两种粒度封禁，按 `logicServerID` 分组存储，支持 per-server 差异化黑名单。

### 7.2 跨进程同步

采用"DB 持久化 + broadcastLogic 广播 + 全量刷新"模式：

```lua
-- 写入 DB 后广播
_script.broadcastLogic("OnMailBlacklistChanged")

-- 收到广播后全量刷新
function ProcessComponent_Mail:OnMailBlacklistChanged()
    self:loadMailBlackList()
    utils.BroadcastToLocalAllAvatars("onMailBlacklistUpdated")
end
```

- 全量刷新而非增量更新，简单可靠，保证最终一致性
- 运行时查询用 hash O(1) 查找，零开销
- 黑名单变更后还会通知在线玩家清理内存中的封禁邮件

## 八、genMail 的字段优先级设计

```lua
-- 三级优先级链：extraInfo > 模板配置 smdd > 默认值
local title   = (extraInfo and extraInfo.title   ~= "" and extraInfo.title) or nil
local content = (extraInfo and extraInfo.content ~= "" and extraInfo.content) or nil
local category = (extraInfo and extraInfo.category) or (smdd and smdd.class) or MAIL_CATEGORY_SYSTEM
```

- 空字符串视为无效值（`~= ""`），防止传入空串覆盖模板配置
- 无模板时（templateId=0）强制要求 title/content/sender 必填，防止空邮件
- 最终构造的 `mailInfo` 是完全独立的 table，不持有对模板或 extraInfo 的任何引用，避免后续修改互相影响

## 九、日志与可观测性

- 邮件快照在 reward 序列化之前生成，保证快照中的 reward 是 Lua 原始格式
- 快照暂存在 `mail._snapshotJson` 上，写 DB 前清除，防止临时字段污染存储
- 批量发送时日志按 50 条分批输出，防止单条日志过长被截断
- DB 写入成功后才上报 SALog，保证日志与实际数据一致

## 总结

这个邮件系统在以下几个关键维度上做得比较到位：

1. **数据一致性** — 先 DB 后内存 + CAS 条件更新 + 失败回滚，多层保障
2. **容错处理** — 三路分支覆盖所有边界，退回机制保证物品不丢失
3. **性能优化** — 微批处理减少 DB 压力，懒加载减少登录开销，防抖避免重复清理
4. **并发控制** — 提取锁 + 超时兜底 + DB 级 CAS，简单有效
5. **运营能力** — 黑名单、撤回、触发邮件等机制完善，支持灵活的运营操作
