## 寻找分线策略
一般发生在玩家申请进入主城（一些偏静态的场景，服务器启动时就创建好了一些分线）
大体逻辑
根据申请进入的人数，单人为1，队伍则看队伍的数量
会从分线列表里面去选取
记分线当前人数+申请人数=人数T
分线列表分了3类
1类是人数T低于softlimit，且分线index小于最小分线数量乘以一个系数，目前是0.5
2类是人数T低于softlimit，分线index大于这个数的
3类是人数T低于softlimit但小于hardlimit的
人数T大于hardlimit的不计入候选列表，这3类内部是平等的，筛选出来之后会shuffle一次
所以如果有1类，就会优先选到其中一个符合条件的1类
2,3类类似
都没有符合条件的时候触发创建分线逻辑。

### 人数计算逻辑
会有预占位的机制，有一个occupyNum来记录已经放行的玩家，同时也会记录放行玩家的AvatarID
在玩家进入场景后，会触发 SEnterWorld
此时会清理occupyNum，增加playerNum
```lua
function Line:reduceOccupyNum(worldID, worldEntityID, avatarID, canEmpty)  
    LOG_INFO("reduceOccupyNum", worldID, worldEntityID, avatarID, canEmpty, self.occupyIds[avatarID])  
    if self.occupyIds[avatarID] then  
        self.occupyIds[avatarID] = nil  
        self.occupyNum = self.occupyNum - 1  
    else  
        if not canEmpty then  
            LOG_ERROR_FMT("reduceOccupyNum occupy not exist %s %s %s", -- [NO_REDMINE]  
                worldID, worldEntityID,avatarID) -- [NO_REDMINE]  
        end  
    endend
```
另外如果玩家进入场景超时（PLAYER_OCCUPY_SEAT_TIME = 30，目前给了30s），也会清理occpuyIds和occupyNum，避免资源一直占用。这里认为超时之后极大概率是玩家没有进入场景，进行清理大概率是正确的逻辑，如果要最精准的人数控制，需要在space加入玩家的之后再加一层保底的判断（可以使用一个略大于hardLimit的阈值），如果还大可以在玩家进入场景的时候把他踢下线。目前看应该没有这个机制。
``` lua
function Line:checkOccupySeatLease(now)  
    local worldID = self.WorldID  
    local worldEntityID = self.worldEntityID  
  
    local occupyIds = self.occupyIds  
    for avatarID, timeout in pairs(occupyIds) do

            LOG_ERROR_FMT("player enter space timeout %s %s %s", -- [NO_REDMINE]  
                worldID, worldEntityID, avatarID) -- [NO_REDMINE]

    endend
```

load的计算，多维度的计算
``` lua
-- 生成一个“平滑负载计算函数”  
-- threshold : 触发惩罚的归一化阈值（如 0.7）  
-- alpha     : 惩罚强度  
-- beta      : 转折陡峭度  
-- weight    : 最终权重  
-- maxValue  : 理论最大值，用于归一化  
-- 返回值：  
--   func(value) -> load  
function GenerateCalcSmoothFunc(threshold, alpha, beta, weight, maxValue)  
    assert(threshold >= 0 and threshold <= 1)  
    assert(alpha >= 0)  
    assert(beta > 0)  
    assert(maxValue > 0)  
    return function(value)  
        -- 归一化  
        local ratio = value / maxValue  
        -- Softplus 平滑惩罚  
        local penalty =  
            alpha * math.log(1 + math.exp(beta * (ratio - threshold)))  
        return (ratio + penalty) * weight  
    end  
end

CalcCpuLoad = GenerateCalcSmoothFunc(  
    config.CPU_THRESHOLD,  
    config.CPU_PENALTY,  
    config.CPU_SHARPNESS,  
    config.LOAD_CPU_WEIGHT,  
    config.PROCESS_MAX_CPU  
)
```

另外在计算load分数时有一个惩罚机制，在阈值之类进行线性增长，在超过某个阈值之后快速增长。
个人理解和一些分类机制类似，例如上面人数的softLimit这种，这里其实在超过阈值之后，基本上要再被分配到那就是大家都超过阈值了，没超过阈值的作为一类，其实会被优先分配到的。

## 负载均衡计算方法

以场景类型算权重：

大世界 9999

50V50 999

其他 10

保证大世界和50V50的均摊，副本消耗较小，基本可忽略
现在似乎没看到这个机制

## 负载均衡3个方案

1.ClusterManager实现一致性算法，将负载选择上报做在ClusterManager上

优势：比较自由拓展       劣势：一致性有待保证

2.将负载选择上报做成单点Service，有挂掉的风险，恢复需要时间

优势：简单可靠              劣势：需要一定时间恢复

3.使用etcd，将负载记录在etcd上，比较可靠，系统复杂度提高，etcd的k、v操作不太支持场景创建的负载操作

优势：较可靠                 劣势：不太能满足当前的业务场景

## 如果使用etcd的几个方案对比

由于mmo的有些场景可能承载上百或上千人，在异步创建的时候，选择负载低的进程时，要先对负载值进行更新再创建，防止并发导致两个大场景分配在一个进程上面，而WorldService横向拓展成多个之后会导致更新抢占问题，需要保证负载更新的原子性

### 方案一   将场景划分为两类-负载高的(大世界、团战)、负载低的（副本、位面），两类负载分别计算

优点：不需要做原子机制(乐观锁等)就可以完成对负载的更新
缺点：WorldService分摊个数较为固定

### 方案二  按照WorldID划分 场景，负载更新做在WorldService上

优点：横向拓展能力强
缺点：需要做原子机制(乐观锁等)保证负载更新的原子性，复杂度提高，系统开销变高

### 方案三  按照WorldID划分 场景，在etcd前架设一个有状态负载更新服务

优点：保证原子性并且复杂度降低，开销降低，能够保证横向拓展能力
缺点：有状态服务相当于是LoadBalancer，如果挂掉场景创建服务将暂不可用，感觉这个etcd也不是很必要，可以做广播采集恢复，用了etcd增加了系统的复杂性，只是换来了恢复的速度

  

## 最终结论

根据方案三最终选定将负载信息单独拆除成一个Service，不使用etcd，依赖进程状态做自动恢复，比较能够适应当前业务的情况

## 流程缺陷1

通知目标分线创建World失败，一直阻塞进入流程，并且占坑失败超时之后会继续阻塞

改造流程
以World为单位上报WorldService心跳
WorldService一段时间内未收到心跳，主动将场景移除

## 流程缺陷2

WorldService挂掉之后，集群状态混乱，需要重启

方案1
WorldID->进程ID+LevelMapID+分线ID(进程自增ID)   问题会造成客户端分线重复
为了保证客户端显示分线ID正确，跟策划沟通分线ID显示规则    100001   100002     1-1   1-2
这样依赖流程缺陷1的心跳保活机制就可以完成重新注册并且不会冲突
策划不希望显示规则更改

方案2
由WorldService分配entityID，WorldService重启之后  重新采集信息，如果上报的WorldID的entity和实际不符，将EntityID相同的给强制销毁

## 流程缺陷3

玩家成功进入场景之后，没有上报进入成功，会导致场景人数越来越多，造成负载不均，进程可能失活

分线playerNum拆分，将其分为正在进入与当前场景人数，方便场景上报进行人数更新，防止因为traceback等问题导致人数一直不更新
