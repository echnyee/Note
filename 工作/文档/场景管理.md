## 寻找分线策略
一般发生在玩家申请进入主城（一些偏静态的场景，服务器启动时就创建好了一些分线）
大体逻辑
根据申请进入的人数，单人为1，队伍则看队伍的数量
会从分线列表里面去选取
记分线当前人数+申请人数=人数T
分线列表分了3类
1类是人数T低于softlimit，且分线index小于最小分线数量乘以一个系数，目前是0.5
2类是人数T低于softlimit，分线index大于这个数的
3类是人数T低于softlimit但小于hardlimit的
人数T大于hardlimit的不计入候选列表，这3类内部是平等的，筛选出来之后会shuffle一次
所以如果有1类，就会优先选到其中一个符合条件的1类
2,3类类似
都没有符合条件的时候触发创建分线逻辑。

### 人数计算逻辑
会有预占位的机制，有一个occupyNum来记录已经放行的玩家，同时也会记录放行玩家的AvatarID
在玩家进入场景后，会触发 SEnterWorld
此时会清理occupyNum，增加playerNum
```lua
function Line:reduceOccupyNum(worldID, worldEntityID, avatarID, canEmpty)  
    LOG_INFO("reduceOccupyNum", worldID, worldEntityID, avatarID, canEmpty, self.occupyIds[avatarID])  
    if self.occupyIds[avatarID] then  
        self.occupyIds[avatarID] = nil  
        self.occupyNum = self.occupyNum - 1  
    else  
        if not canEmpty then  
            LOG_ERROR_FMT("reduceOccupyNum occupy not exist %s %s %s", -- [NO_REDMINE]  
                worldID, worldEntityID,avatarID) -- [NO_REDMINE]  
        end  
    endend
```
另外如果玩家进入场景超时（PLAYER_OCCUPY_SEAT_TIME = 30，目前给了30s），也会清理occpuyIds和occupyNum，避免资源一直占用。这里认为超时之后极大概率是玩家没有进入场景，进行清理大概率是正确的逻辑，如果要最精准的人数控制，需要在space加入玩家的之后再加一层保底的判断（使用一个略大于hardLimit的阈值），如果还大可以在玩家进入场景的时候把他踢下线。目前看应该没有这个机制。
``` lua
function Line:checkOccupySeatLease(now)  
    local worldID = self.WorldID  
    local worldEntityID = self.worldEntityID  
  
    local occupyIds = self.occupyIds  
    for avatarID, timeout in pairs(occupyIds) do  
        if now > timeout then  
            self:reduceOccupyNum(worldID, worldEntityID, avatarID)  
            LOG_ERROR_FMT("player enter space timeout %s %s %s", -- [NO_REDMINE]  
                worldID, worldEntityID, avatarID) -- [NO_REDMINE]  
        end  
    endend
```

load的计算，多维度的计算
``` lua
-- 生成一个“平滑负载计算函数”  
-- threshold : 触发惩罚的归一化阈值（如 0.7）  
-- alpha     : 惩罚强度  
-- beta      : 转折陡峭度  
-- weight    : 最终权重  
-- maxValue  : 理论最大值，用于归一化  
-- 返回值：  
--   func(value) -> load  
function GenerateCalcSmoothFunc(threshold, alpha, beta, weight, maxValue)  
    assert(threshold >= 0 and threshold <= 1)  
    assert(alpha >= 0)  
    assert(beta > 0)  
    assert(maxValue > 0)  
    return function(value)  
        -- 归一化  
        local ratio = value / maxValue  
        -- Softplus 平滑惩罚  
        local penalty =  
            alpha * math.log(1 + math.exp(beta * (ratio - threshold)))  
        return (ratio + penalty) * weight  
    end  
end

CalcCpuLoad = GenerateCalcSmoothFunc(  
    config.CPU_THRESHOLD,  
    config.CPU_PENALTY,  
    config.CPU_SHARPNESS,  
    config.LOAD_CPU_WEIGHT,  
    config.PROCESS_MAX_CPU  
)

LOAD_BALANCE_CONFIG = {  
    SELECT_LOWEST_TOP_N = 10,  
  
    LOAD_MEM_WEIGHT = 100,  
    LOAD_CPU_WEIGHT = 100,  
    LOAD_GC_WEIGHT = 100,  
    LOAD_PLAYER_NUM_WEIGHT = 100,  
    PROCESS_MAX_CPU = 100,  
    PROCESS_MAX_PLAYER_NUM = 400,  
    PROCESS_MAX_GC = 1024,  
    PROCESS_MAX_MEM = 1536,  
  
    CPU_THRESHOLD = 0.7,  
    CPU_PENALTY = 0.5,  
    CPU_SHARPNESS = 15,  
    MEM_THRESHOLD = 0.8,  
    MEM_PENALTY = 0.0,  
    MEM_SHARPNESS = 10,  
    PLAYER_THRESHOLD = 0.8,  
    PLAYER_PENALTY = 0.3,  
    PLAYER_SHARPNESS = 10,  
  
    GC_THRESHOLD = 0.8,  
    GC_PENALTY = 0.0,  
    GC_SHARPNESS = 10,  
}

```

### 一些疑问
目前的策略里面，在5s内如果有大量创建场景的请求，看起来就会在当前10个最低负载的logic上创建，没有看到预占位性能的逻辑，直接使用场景对应的占用是不是会更好一点，首先现在有5s的延迟，在没有新的性能指标上传之前容易造成大量场景创建在某几个logic上，另外一些场景在创建初期不一定会有明显的性能开销，此时获取到的cpu并不能真正反应之后未来的情况。
一些可能的原因，某个logic上的开销不光是战斗副本，可能会有其他的服务，这样可能更通用一些。然后短时间大量创建副本的场景相对较少。
不过如果这个logic上就只开战斗副本，那就可以通过预设的副本占用预估来进行负载均衡，可以和人数类似，也使用预占位之类的机制，这样可以支持短时间大批量创建副本的场景。
