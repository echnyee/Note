# 快速模式指南（C7 游戏服务端）

> 快速模式是标准 Spec Coding 流程的精简版本，直接分析需求生成合并文档后进入开发。

---

## 1. 什么是快速模式

快速模式适用于：
- 需求明确、边界清晰的小功能
- Bug 修复
- 配置调整、参数修改
- 用户对流程已熟悉，不需要逐步确认

**核心区别**：

| 维度 | 标准模式 | 快速模式 |
|------|----------|----------|
| 文档数量 | 分离的需求/技术/测试文档 | 合并为单一文档 |
| 确认环节 | 每步确认 | 最多 3 个问题，一次确认 |
| 适用场景 | 复杂功能/新系统 | 小功能/Bug 修复 |

---

## 2. 触发场景

当用户说以下内容时触发快速模式：
- "快速实现一下 XXX"
- "直接开始写代码"
- "小功能，直接做"
- "修 Bug"
- "快速模式"
- "不用写那么多文档"

---

## 3. 快速模式工作流

```
用户描述需求
        ↓
Step 1: 需求理解
        - AI 快速澄清关键问题
        - 最多 3 个确认问题
        ↓
Step 2: 生成合并文档
        - 生成 F00X-quick-spec.md
        - 包含需求 + 技术方案 + 测试要点
        ↓
Step 3: 用户确认
        - 展示文档摘要
        - 用户说"可以"即开始
        ↓
Step 4: 直接实现
        - p4 edit 相关文件
        - 生成 TODO 清单
        - 逐个实现
        ↓
Step 5: 验证
        - 通过 telnet 或 GM 命令验证
        ↓
完成
```

## 4. 快速确认问题模板

```
收到！让我快速确认几个关键点：

1. **核心功能**：[AI 理解的核心功能]，对吗？
2. **边界情况**：[AI 推测的边界处理]，这样处理可以吗？
3. **涉及范围**：我看需要修改 [文件列表]，确认一下？

确认后我直接开始。
```

**规则**：最多问 3 个问题，用户说"可以/对/开始"即进入实现。

## 6. 何时用快速模式

**适用**：
- 修复一个 Bug
- 添加一个小功能点（如新增一个 GM 命令）
- 修改配表读取逻辑
- 调整数值参数
- 给现有 Component 添加一个方法
- 需求方已经想清楚了

**不适用**：
- 全新的游戏系统
- 需求不明确
- 涉及多个 Entity/Component/Service 联动
- 新增 Entity 类型
- 涉及跨服/跨进程交互设计
- 需要团队评审的改动

---

## 7. 快速模式实现原则

### 7.1 推荐做法

- 快速确认，不要反复问
- 合并文档，不要分散写
- p4 edit 后直接开始实现
- 实现后通过 telnet/GM 验证
- 关键信息记录完整

### 7.2 禁止做法

- 完全不写文档（至少要有 quick-spec.md）
- 不做任何确认就开始
- 跳过验证
- 不对照 system-dev-checklist 检查

---

## 8. 从快速模式切换到标准模式

如果实现过程中发现需求比预期复杂：

```
实现过程中我发现这个功能比预期复杂：
- [复杂点 1]
- [复杂点 2]

建议切换到标准模式，生成完整的需求和技术文档。
确认切换？
```
