- 当有多种实现方案时，询问用户偏好

- 多进程游戏服务器，引擎层（C++）提供基础服务，业务层使用 Lua 脚本开发，运行在 logic 进程。
- 在讨论任何项目问题或进行任何代码修改前，必须先完整阅读 `../docs/00-core/project-framework.md` 

## Perforce 工作流（强制）

项目由 Perforce 管理。为避免 P4 命令阻塞编辑流程，采用**先改后提**策略：

### 编辑阶段：直接修改，不做 P4 操作

- 修改已有文件时，**跳过 `p4 edit`**，直接用 `attrib -r` 去掉只读属性后修改文件
- 创建新文件时，直接创建，不执行 `p4 add`
- 此阶段**禁止执行任何 P4 命令**，保证编辑流程零等待

### 收尾阶段：所有修改完成后统一执行 P4 操作

当所有文件修改全部完成后，**一次性**执行 P4 命令：

1. 对所有**修改过的已有文件**批量 checkout：`p4 edit file1 file2 file3 ...`
2. 对所有**新建的文件**批量添加：`p4 add file1 file2 file3 ...`
3. 不需要向用户确认，直接执行

### P4 命令超时与重试策略

收尾阶段的 P4 命令**必须设置 timeout 参数**：

| 命令类型 | timeout（毫秒） |
|----------|:---:|
| 轻量操作：`p4 edit`、`p4 add`、`p4 opened` | **15000** |
| 中量操作：`p4 diff`、`p4 diff2`、`p4 print`（单文件） | **30000** |
| 重量操作：`p4 print`（大文件）、`p4 unshelve`、`p4 shelve` | **60000** |

**重试规则**：
- 超时后**最多自动重试 2 次**（共 3 次尝试）
- 3 次均失败后报告超时，建议用户检查网络或 P4 服务器状态
- **不要因为单个 P4 命令超时就停止整个工作流**

## 编码规范
- 语言：Luajit 2.1
- 注释使用中文 使用 EmmyLua 风格的类型注解（`---@param`、`---@return`）
- 网络协议定义在 `NetDefs/` 目录的 XML 文件中，修改组件RPC函数时需要同步更改
- **编写或修改 Lua 代码前**，必须先阅读 `../docs/00-core/lua-code-style.md`，严格遵守其中的所有编码规范
- **强制**以资深服务端工程师标准要求自己：不仅实现功能正确，还必须考虑性能细节和可维护性
- 不要写冗余兼容代码，除非我明确要求
