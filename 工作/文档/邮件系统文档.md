# 邮件系统服务端文档

## 一、系统架构

邮件系统采用两层架构：

- **进程层** `ProcessComponent_Mail.lua` — 挂载在 Process 上，负责邮件生成、DB 写入、全服邮件管理、触发邮件定时器、邮件黑名单
- **玩家层** `MailComponent.lua` — 挂载在 AvatarActor 上，负责玩家侧的邮件查看、删除、附件提取、全服邮件接收、触发邮件条件检查
- **工具层** `MailUtils.lua` — 公共辅助函数，GM 发送接口、触发邮件错误码、邮件快照等

```
┌─────────────────────────────────────────────────────┐
│                    Process 进程                      │
│  ProcessComponent_Mail                              │
│  ├── genMail()           邮件内容生成                │
│  ├── sendAvatarActor*    个人邮件发送队列             │
│  ├── sendServer*         全服邮件发送                │
│  ├── setupTriggerMailTimer  触发邮件定时器            │
│  ├── cleanOverLimitMails    超限清理                 │
│  └── 邮件黑名单管理                                  │
├─────────────────────────────────────────────────────┤
│              AvatarActor 玩家实体                     │
│  MailComponent                                      │
│  ├── getMailList / getMail     邮件查询              │
│  ├── deleteMail / batchDelete  邮件删除              │

│  ├── loginCheckMailTrigger     触发邮件检查           │
│  └── 邮件收藏管理                                    │
└─────────────────────────────────────────────────────┘
```

## 二、邮件类型

| 类型 | 说明 | 存储位置 |
|---|---|---|
| 个人系统邮件 | 按模板发送给指定玩家，templateId > 0 | `mail` 集合（按 userID） |
| 个人自定义邮件 | templateId=0，title/content/sender 由调用方指定 | `mail` 集合 |
| 全服邮件 | 写入共享表，玩家登录或收到通知后拉取 | `server_mail` 集合 |
| 触发邮件 | 配置了触发条件（时间/等级/自定义触发器），条件满���后自动发送 | 模板配置 + `mail_trigger` 集合 |

邮件内容类型：
- `MAIL_TYPE_TEXT` — 纯文本邮件（无附件）
- `MAIL_TYPE_ATTACHMENT` — 带附件邮件

邮件分类：
- `MAIL_CATEGORY_SYSTEM` — 系统邮件（默认）
- `MAIL_CATEGORY_GUILD` — 公会邮件

## 七、核心流程

### 7.1 发送个人邮件

```
外部系统调用
  → Process:sendAvatarActorSystemMail(templateId, avatarId, reward, ...)
    → genMail() 生成邮件对象
    → _sendAvatarActorMail() 分配UUID
    → addSendAvatarActorMailTask() 加入发送队列
    → 0.1s后 processAvatarActorMailTask() 批量处理
      → 按 logicServerID 分组
      → 每100封一批 MongoBulk upsert 写入DB
      → 上报 SALog
      → 通知在线玩家 notifyMail
```

### 7.2 发送全服邮件

```
GM/运营后台调用
  → Process:sendServerCustomMail(reward, conditionInfo, ...)
    → genMail() 生成邮件对象
    → IncAndFetchSharedCounter("server_mail_id") 获取全局递增 serverMailId
    → 写入 SERVER_MAIL 集合
    → broadcastLogic("OnNewGlobalMail") 通知所有逻辑进程
      → BatchLoadServerMail() 从 DB 加载未过期全服邮件到 Game.Servermails
      → 各进程通知在线玩家 checkServerMail
        → 玩家检查条件（等级/建号时间/发送时间窗口/黑名单）
        → 满足条件且 receivedServerMailIds 中未记录 → 创建邮件到个人邮箱
        → 不满足条�� → 跳过但不标记，后续条件满足时仍可接收
```

#### serverMailId 的生成机制

`serverMailId` 通过 `SHARED_COUNTER` 集合中的 `server_mail_id` 计数器原子自增生成，而非使用 `SERVER_MAIL` 集合自身的 `_id`。原因：

- MongoDB 默认 `_id` 是 ObjectId（字符串），不是递增数字，无法直接做大小比较
- `IncAndFetchSharedCounter` 使用 `MongoFindAndUpdate` + `$inc` 原子操作，多进程并发时保证唯一且递增
- 先拿号再写入的流程比写入后取号更可控，`serverMailId` 在写 DB 前就已确定

#### 玩家侧接收记录：receivedServerMailIds

玩家使用 `receivedServerMailIds`（map<number, boolean>）逐条记录已接收的全服邮件 `serverMailId`，而非单一水位线 `lastServerMailId`。原因：

全服邮件存在条件过滤（等级/建号时间/生效时间/黑名单），玩家可能暂时不满足某封邮件的条件。假设邮件序列为 1,2,3,4,5，玩家当前只满足 1,3,5 的条件，若用水位线 `lastServerMailId=5` 一刀切，后续等级提升后 2 和 4 就永远收不到了。逐条记录方式下，跳过的邮件不会被标记，下次 `checkServerMail` 时重新检查条件，满足了就能收到。

#### 全服邮件内存缓存（Game.Servermails）

全服邮件通过 `BatchLoadServerMail` 从 DB 加载到进程内存 `Game.Servermails[logicServerID][serverMailId]` 中。玩家的 `checkServerMail` 直接读内存而非查 DB，将 N 个玩家的 N 次 DB 查询降为 0。加载时只取未过期邮件，用游标分页递归加载避免单次查询过大。

#### 分帧广播通知在线玩家

全服邮件到达后通过 `BroadcastToLocalAllAvatars("checkServerMail")` 通知所有在线玩家。该函数采用分帧处理：每批 100 个玩家，批次间隔 100ms，避免一帧内遍历所有在线玩家导致 CPU 尖峰。每个玩家的调用用 `safeCall` 保护，单个异常不影响其他玩家。

### 7.3 附件提取

```
客户端请求 extractMail(mailId)
  → 检查邮件是否存在、是否有附件、是否已领取
  → 检查邮件是否被封禁（黑名单）
  → 检查玩家是否被邮件封禁（BAN_MAIL）
  → setExtractMailLock() 加提取锁
  → 预检查背包容量
  → DB 标记邮件为已领取
  → 物品进背包（BagComponent:addItemList）
    → 成功：通知客户端 onExtractMail
    → 部分成功：放不下的物品以新邮件退回（_sendRemainderMail）
    → 全部失败：回滚DB状态
  → unsetExtractMailLock() 释放锁
  → ScheduleCleanOverLimitMails() 调度超限清理
```

### 7.4 触发邮件

```
玩家登录
  → loginCheckMailTrigger()
    → 遍历所有触发邮件模板
    → 检查静态条件（时间窗口/等级/建号时间）
    → 条件满足 → 直接发送
    → 条件不满足但未过期 → 注册触发器（等级变化等）

Process 定时器到期
  → onTriggerMailTimer(templateID)
    → broadcastLogic 通知所有在线玩家
      → checkTriggerMailOnline() 重新检查条件

触发器条件完成
  → MailOnTriggerConditionComplete()
    → _tryActiveTriggerMail() 尝试激活并发送
```

### 7.5 邮件收藏

```
客户端请求 ReqMailFavAdd(mailId)
  → 检查邮件是否存在
  → 检查收藏数量是否达上限（100）
  → 检查是否已收藏
  → 写入 MAIL_FAVORITES 集合
  → 返回 RetMailFavAdd

模板配置 addToFav=true 的邮件
  → 创建时自动调用 _autoMailFavAdd() 收藏
```

## 八、邮件过期与清理

- 默认过期时间 30 天，带附件邮件永不过��
- 模板 `ValidityPeriod` 字段可自定义过期天数（-1=永不过期）
- 模板 `isDelete` 字段控制删除策略（0=不可删除，3=可删除等）
- 玩家邮件超过 200 封时，自动清理最旧的纯文本邮件和已领取附件邮件
- 清理通过 `ScheduleCleanOverLimitMails` 调度，1s 延迟防重复

## 九、邮件黑名单

- 支持按模板ID封禁邮件，封禁后该模板邮件不可提取附件
- 黑名单存储在 `mail_blacklist` DB 集合
- 操作后通过 `broadcastLogic` 跨进程同步
- 接口：`AddMailToBlacklist` / `RemoveMailFromBlacklist` / `IsMailInBlacklist` / `GetMailBlacklist`

## 十、邮件撤回

- 通过 `DeleteMailsByOpNUID(logicServerID, opNUID, deleteMaxLimit, callback)` 实现
- 按 opNUID 批量删除未领取的邮件
- 有数量上限保护，防止误操作影响范围过大
